# Skill Creator

用于在当前项目的 agent 工作区中创建、修改、校验技能文件。  
当用户提到“创建 skill / 新增 skill / 修改 skill / 优化 skill / 更新 SKILL.md / 调整技能规则”时使用本技能。

## 目标

- 在项目内稳定创建新 skill（`skills/<skill-name>/SKILL.md`）
- 在不破坏原有能力的前提下修改已有 skill
- 确保 skill 写入后可以被系统发现并读取

## 必须遵守

- 仅在当前工作区的 `skills/` 下操作，不写入 `~/.cursor/skills-cursor/`
- skill 名必须使用小写英文与中划线（如 `essay-grading-html`）
- 若用户提供“必须原文保留”的句子，写入时不得改写
- 修改已有 skill 时，除用户要求变更的部分外，尽量保持原结构与语义稳定

## 创建新 Skill（强制流程）

1. 收集最小需求（缺一则先问）：
   - skill 名称
   - 解决什么问题
   - 何时触发
   - 关键执行步骤
   - 输出格式要求
2. 检查 `skills/` 下是否已有同名目录：
   - 若存在：进入“修改已有 Skill”流程
   - 若不存在：创建 `skills/<skill-name>/`
3. 写入 `skills/<skill-name>/SKILL.md`，至少包含：
   - 标题
   - 一段“用途+触发时机”说明
   - 分步骤执行流程
   - 输出约束
   - 失败回退策略
4. 写完后立即自检：
   - 读取回写文件确认内容完整
   - 关键术语前后一致
   - 路径、工具名、输出文件名无冲突

## 修改已有 Skill（强制流程）

1. 先读取原 `SKILL.md`，提取：
   - 当前流程
   - 输出结构
   - 关键硬约束
2. 按用户要求做“最小必要改动”：
   - 新增要求：补充到对应章节并加硬约束
   - 删除要求：仅删除指定能力，不误删其他流程
   - 调整顺序：保留依赖关系（先决步骤不能被提前）
3. 修改后自检：
   - 是否仍可执行（步骤闭环）
   - 是否有前后矛盾条款
   - 是否遗漏用户新增的硬要求

## 推荐 SKILL.md 模板

```markdown
# <skill-name>

用于处理 <任务类型>。当用户提到 <触发词> 时使用本技能。

## 输入要求
- ...

## 强制流程
1. ...
2. ...
3. ...

## 输出要求
- ...

## 失败与回退
- ...
```

## 可发现性校验（必须执行）

完成创建/修改后，至少执行一项校验：

- 方式 A：通过技能列表确认出现 `skill-name`
- 方式 B：调用 `read_skill` 读取该 skill，确认内容可访问

若校验失败：

1. 先检查路径是否为 `skills/<skill-name>/SKILL.md`
2. 检查文件名是否精确为 `SKILL.md`
3. 检查编码与内容是否为空

## 对用户的交付格式

最终回复必须包含：

- skill 名称
- 实际文件路径
- 本次是“新建”还是“修改”
- 关键变更点（3-6 条）
- 建议触发语（1 句）

## 常见错误与修复

- 错误：把 skill 写到错误目录  
  修复：移动到 `skills/<skill-name>/SKILL.md`
- 错误：修改时覆盖了用户旧规则  
  修复：回读旧版，按“最小必要改动”重写
- 错误：写完但系统发现不到  
  修复：重查目录层级与文件名大小写，重新校验 `read_skill`
